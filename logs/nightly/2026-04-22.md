## 2. What's worth noticing

**Two phases in two days is a velocity worth naming.** Phase 1
closed 2026-04-21 evening, Phase 2 closed 2026-04-22 late
morning. Combined output: ~1,500 lines of new Python across the
MMF module, real atomic DvP on XRPL testnet, full persistence,
full validation matrix. Sandbox + testnet-only discipline is
what makes this velocity safe — no real money, no real investor
keys (yet), no mainnet exposure. When P3-5 flips the custody
model to investor-signed-tx and P3-6 writes the mainnet-readiness
doctrine, velocity has to slow — those changes touch the actual
trust boundary. Two days for Phase 1-2 is the sandbox rate;
Phase 3-4 will not and should not match it.

**tecPATH_DRY was the Phase 2 lesson about writing specs ahead of
live testing.** My own Phase 2 brief said "expected tecNO_AUTH"
for BREAK 8. XRPL actually returned tecPATH_DRY because cross-
currency Payment uses pathfinding, and pathfinding encounters
an unauthorized trust line as a dead path, not a direct
authorization failure. Same semantics, different engine code.
The brief should have said "expected tec* failure consistent
with unauthorized-line rejection" — leaving the specific code
to be validated against live behavior. Writing the exact
expected code without having observed it is a prediction that
can be wrong, not a fact that can be checked. General lesson:
when spec'ing expected XRPL (or any external-ledger) response
codes in a test matrix, bound the expectation to the failure
class, not to a specific literal.

**Shipped Lane D HITL sync when the brief said async.** Phase 2
P2-2's brief called for async HITL with a separate
`DIGITAL_SUBSCRIPTION_DVP_APPROVED` event between submission and
atomic execution. Implementation shipped synchronous — the
`/api/mmf/subscribe` request blocks for ~20 seconds while the
DvP runs in-line. Flagged this in the P2-2 report at the time;
carried forward as P3-4 in Phase 3 brief. Worth naming because
this is a doctrine deviation from my own brief, not from the
operator's. Happy-path sync vs async is the difference between
"operator sent the request IS the approval" and "operator
reviewed and approved AFTER the system flagged pending." For a
sandbox testnet the sync shortcut is fine. For real investor
flows it isn't — which is why P3-4 promotes async HITL to a
first-class prompt rather than a nice-to-have.

**Persistence's quietest lesson: the hydrate-from-disk default is
a sneaky source of confusion.** My nav_engine's
`_hydrate_from_disk` returns silently if the state file doesn't
exist. The module then runs with its DEFAULT state (sweep_count=0,
cnav/fnav=$1.0000, etc.). This is indistinguishable from "the
file existed and loaded a zero state" at the observability layer.
I flagged this as a possible P2-1 bug in my P2-2 report and
didn't close the loop until the diagnostic today. The code was
fine; my mental model was wrong. Default state looks like
persisted state when nothing has been written yet. The fix
isn't in the code — it's in the operator observability. A
`"hydrated": true/false` flag on `get_nav_state()` output would
have made this unambiguous from a single curl. Adding that is
Phase 3 polish.
