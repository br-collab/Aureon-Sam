3. **userref / cl_ord_id.** Kraken `AddOrder` takes `userref`
   (32-bit int), not Gemini-style `cl_ord_id`. Every order was
   rejected with `EGeneral:Invalid arguments:cl_ord_id`. Fixed via
   `int(sha256(client_order_id.encode()).hexdigest(), 16) % 2**31`.
4. **#2 DSOR encoder.** `gate_records` and `dsor_entries` contain

5. **#3 Kill-switch AttributeError.** `ThifurH.kill_switch()` called

   `GeminiSandboxClient`. `KrakenLiveClient` had `cancel_all_orders`.
   Added an alias method on the Kraken client. (`2c7f778`)
6. **#7 ORDER_ERROR false-positive.** `if "error" in order_response`
   is a key-presence check; Kraken responses *always* contain an
   `"error"` key (empty list on success, populated on failure).
   Every successful order was being recorded as `ORDER_ERROR`,
   `orders_placed` never incremented, `open_positions` never
   tracked. Real txid `O2IKZW-V56UJ-3I5QO6` was placed live before
   I caught it — bug #3 fix had just shipped, so kill-switch
   actually cancelled it. Fixed via `if order_response.get("error")`.
   (`d5b9881`)
7. **#8 Price precision.** Kraken BTC/USD requires max 1 decimal
   (tick size $0.10). Signal was rounding to 2 decimals; orders
   intermittently failed with `EOrder:Invalid price`. Earlier
   successful order slipped through by luck (price * 0.999 happened
   to land cleanly). (`0f4cd7c`)
8. **#9 Order ID extraction.** `order_response.get("order_id",
   "UNKNOWN")` is the Gemini path; Kraken returns it at
   `result.txid[0]`. `open_positions` was being keyed under literal
   `"UNKNOWN"`. Multi-order sessions would collide. Fixed.
   (`6cf5b7b`)
9. **#4 Atrox rationale doctrine leak + #6 gate harness short-

   the actual order. Stripped from Atrox; signal route rewrites

## 2. What's worth noticing

**Discipline lives at the lowest enforceable layer, not the
highest.** I built `MAX_CONCURRENT_POSITIONS` as a soft check
inside the Atrox Live loop's BUY-emission logic, treating the
loop as the only signal source. The manual `/signal` endpoint
exists in parallel and has no such check. The operator clicked
the button six times and immediately blew past the doctrine. The
fix isn't loop logic — it's a gate (Gate 6, or extend Gate 4) in
the harness every signal must pass through. Recurring pattern:
when a constraint is "important," the place to put it is the
narrowest choke point, not the most convenient one.

**The kill switch label was wrong.** I called it "the operator's
last-resort control" and "the hard floor" multiple times today —
in commits, chat, README. It cancels *resting orders*; it does
not *flatten*. With six filled positions and ZUSD at $21, the
operator hitting kill-switch alone would have left $45 of long
BTC exposure orphaned (in-memory `open_positions` cleared,
Kraken still long). The label drove false confidence. Worth
flagging that the words I use for safety mechanisms shape what
the operator believes about their coverage. Lesson: name things
by what they *do*, not by what role I want them to play.

**The "let it run and learn" choice came from the operator, not
me.** When the breach surfaced my instinct was to recommend
liquidate-and-reset (clean state). The operator pushed back with
"failure is learning" and reframed to "what would you do." On
re-think the right move was option 2 (arm policy, drain the
positions, validate the auto-execution path under real
conditions) — exactly because the test that's running is the
test we hadn't run yet (`process_close_signal` had never fired
against live Kraken). I had to be told. The sandboxed-test
mindset wants to start over; the production-pedagogy mindset
keeps the run.

**Aureon-Leto evolved as a clean staircase, not a rewrite.** From
file-queue bridge → Anthropic API auto-dispatch → context
injection → markdown rendering → dismiss buttons → rebrand. Each
step preserved the previous. The file-queue bridge still works
with no API key set; the API path doesn't replace it, layers on
top. That additive-not-replacement pattern survives a lot of
operator-driven iteration without drift. Worth keeping.

**`<situational_context>` injection lands.** First context-aware
Sam response (via Anthropic API) gave a clean status briefing —
no prompting, no "what's the cycle, what's the balance" hand-
feeding. The operator immediately escalated to "every dispatch
should have this." Validates that structuring state into XML-
style tags inside the user message (vs jamming into the system
prompt) lets Claude separate identity-of-role from situational
data. Slightly more tokens (~1.2K input per task vs ~0.4K
without context) but the response quality jump is large.

**Two confidence-vs-correctness gaps I shipped today.** (a) Kill-
switch label as above. (b) `MAX_CONCURRENT_POSITIONS` as
loop-logic. Both were architectures I was confident about while
designing; both turned out to be wrong at the layer I'd placed
them. The pattern is: I tend to enforce constraints at the layer
where they're *expressed*, not at the layer where they're
*violable*. Keep watching for this.
